## What was built this session

### 1. Docker deployment (`docker-compose.yml` + two Dockerfiles)

**Problem:** No containerised deployment existed. Running the project required manually installing Python 3.14, uv, Node 20, Ollama, and pulling four models.

**Solution:**

| File | Purpose |
|---|---|
| `backend/Dockerfile` | Python 3.14-slim + uv; copies pipeline code; runs entrypoint script |
| `backend/docker-entrypoint.sh` | Downloads `all-MiniLM-L6-v2` embedding model on first start if not present |
| `frontend/Dockerfile` | Multi-stage Node 20 build — deps → build (with `BACKEND_URL` baked in) → runtime |
| `docker-compose.yml` | Orchestrates both services; volume-mounts `backend/pipeline/data/` for large CVE files |
| `.env.example` (root) | Documents all env vars needed for Docker Compose |

**Usage:**
```bash
cp .env.example .env
# Fill in OPENROUTER_API_KEY (or use LLM_BACKEND=ollama for local GPU)
docker compose up --build
# Frontend → http://localhost:3000
# Backend  → http://localhost:8000
```

Large data files (`cve_embeddings_local.npz`, `nvd_cves_min.jsonl`) are volume-mounted — not baked into the image. The embedding model (~90 MB) is downloaded automatically by `docker-entrypoint.sh` on first start.

### 2. OpenRouter cloud backend (`pipeline/llm/openrouter_client.py`)

**Problem:** The pipeline was tied to a local Ollama instance, requiring a GPU. Users without one had no option.

**Solution:** New `openrouter_client.py` — a drop-in replacement that routes all `ollama_chat` / `ollama_generate` calls through OpenRouter's OpenAI-compatible API. No changes to any pipeline stage.

**Key design:**
- Streaming with deadline-based cancellation (same `LLMTimeoutError` as the Ollama client)
- **Per-model-group API keys** for parallel rate limits:

| Env var | Models | Stages |
|---|---|---|
| `OPENROUTER_API_KEY_REASONING` | deepseek / r1 | preprocessing, RAG |
| `OPENROUTER_API_KEY_CODING` | qwen / coder | recommender, verifier |
| `OPENROUTER_API_KEY_INSTRUCTION` | llama | validator |
| `OPENROUTER_API_KEY_SUMMARIZE` | mistral | reporter |
| `OPENROUTER_API_KEY` | any | master fallback |

With separate keys each stage gets its own OpenRouter rate-limit bucket — all stages can run concurrently without throttling each other.

**Model name mapping** in `config.yaml`:
```yaml
openrouter:
  model_map:
    "deepseek-r1:8b":   "deepseek/deepseek-r1-distill-llama-8b"
    "llama3.1:8b":      "meta-llama/llama-3.1-8b-instruct"
    "qwen2.5-coder:7b": "qwen/qwen-2.5-coder-7b-instruct"
    "mistral:7b":       "mistralai/mistral-7b-instruct"
```

Switching to OpenRouter: set `LLM_BACKEND=openrouter`. Switching back: unset it or set `LLM_BACKEND=ollama`. No other code changes.

### 3. Per-request backend selection (frontend + backend)

**Problem:** `LLM_BACKEND` was a server-wide env var — every user of a shared deployment had to use the same backend.

**Solution:** Users can now choose their backend per-session from the UI. The choice travels with each `/analyze` request and is applied thread-safely using Python `contextvars`.

**Backend — `pipeline/llm/context.py`:**
```python
REQUEST_BACKEND: ContextVar[str]   # "ollama" | "openrouter" | ""
REQUEST_OR_KEY:  ContextVar[str]   # per-request OpenRouter API key
```
`ThreadPoolExecutor.submit()` copies the context to child threads (PEP 567), so values set in `_run_pipeline` are automatically inherited by the parallel function-processing threads.

**`AnalyzeRequest` new fields:**
```python
backend: str | None = None           # per-request backend override
openrouter_api_key: str | None = None # user-supplied key (never stored server-side)
```

**Frontend — `BackendSelector` component in the toolbar:**

Three option cards in a neumorphic dropdown panel:

| Card | Headline |
|---|---|
| `Cpu` | "Have the hardware? Try local LLMs" |
| `Cloud` | "Have your own API key? Use here" + key input |
| `Settings2` | "Use server default" |

When "Cloud" is selected, a master API key input appears plus an expandable "Advanced" section for per-stage group keys.

User preference and key are persisted to `localStorage` via Zustand `persist` middleware — no re-entry needed on page reload.

**`ConnectionPill` in toolbar is now backend-aware:**
- OpenRouter + no key → amber "No API key"
- OpenRouter + key present → "Cloud API"
- Ollama + Ollama offline → existing amber "Ollama offline"
- Ollama + online → "Ready"

### 4. Parallel function processing (`server.py`)

**Problem:** Functions in a file were processed sequentially — a file with 5 functions took 5× the slowest function's time.

**Solution:** Replaced the sequential `for` loop in `_execute_pipeline` with a `ThreadPoolExecutor`. All functions now run their full RAG → validate → risk → recommend pipeline in parallel. Report writes are serialised with a `threading.Lock()` because `FileStore` is a plain dict.

```yaml
# config.yaml — new setting
settings:
  max_function_workers: 4  # raise to 8+ when using OpenRouter
```

With Ollama: the GPU semaphore still serialises actual LLM calls — parallelism helps by overlapping pre/post-LLM work across functions.  
With OpenRouter: true parallel cloud inference — all functions advance simultaneously.
