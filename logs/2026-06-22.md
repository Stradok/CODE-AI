## Issues found and fixed

### 1. CVE detection stopped working (RAG broken)

**Symptom:** Same SQL injection code that detected vulnerabilities before now returned nothing.

**Root cause:** `pipeline/data/cve_embeddings_local.npz` was a **directory** instead of a file — the `.npz` archive had been unzipped in-place. `np.load()` hit `IsADirectoryError`, which was silently caught by `_prewarm_rag()`, leaving the embedding index never loaded. Every analysis returned zero CVEs.

**Contents of the corrupt path:**
```
pipeline/data/cve_embeddings_local.npz/
  embeddings.npy   (259 MB — 168,960 × 384 float32)
  ids.npy          (10 MB  — 168,960 CVE ID strings)
```

**Fix:** Repacked the two `.npy` files into a proper `.npz` archive:
```python
ids = np.load("pipeline/data/cve_embeddings_local.npz/ids.npy", allow_pickle=True)
embeddings = np.load("pipeline/data/cve_embeddings_local.npz/embeddings.npy", allow_pickle=True)
np.savez_compressed("pipeline/data/cve_embeddings_repacked.npz", ids=ids, embeddings=embeddings)
```
Then `rm -rf` the directory and moved the file into place.

**Verification:** RAG retrieval on SQL injection code returns 6 CVEs with scores 0.38–0.44 (well above the 0.25 threshold).

### 2. LLM calls timing out after 300s instead of 120s — 30-minute stalls

**Symptom:** RAG analysis stage (`vulnerable_login` function) timed out after 300s. Total run took ~30 minutes and still failed.

**Root causes (three compounding bugs):**

| # | Bug | Effect |
|---|-----|--------|
| 1 | `ollama_chat/ollama_generate` had `timeout: int = 300` hardcoded — never read `settings.llm_timeout` from config | 300s wait instead of 120s |
| 2 | `thread.join(timeout)` stopped *waiting* but the Ollama request kept running on the GPU | Every retry hit a busy server → also timed out |
| 3 | `with_retry` retried `LLMTimeoutError` 3× full timeout each time | 3 retries × 300s × 2 stages ≈ 30min |

**Also noticed:** The log showed `llama3.1:8b` being used for the RAG stage. `llama3.1:8b` is the fast instruction model (meant for the validator). The RAG stage needs `deepseek-r1:8b` for complex chain-of-thought CVE analysis. Using the wrong model caused JSON parse failures → all 3 retries exhausted. The user had selected it via the model selector UI — resetting to defaults fixes it.

**Fix — `pipeline/llm/ollama_client.py`:**
- Replaced thread-based timeout with **streaming API + deadline loop**
- Closing the stream (`stream.close()`) cancels the Ollama request server-side — no zombie generation
- Timeout now read from `_cfg_timeout()` → `get_settings()["llm_timeout"]` (120s)
- Added `num_predict` cap: `chat=4096`, `generate=1024` — prevents runaway token generation

**Fix — `pipeline/llm/retry.py`:**
- `LLMTimeoutError` now gets **at most 1 retry** (not 3). If it times out twice, fail fast.

### 3. Race condition + GPU thrashing on hardware-limited system

**Symptom:** User reported hardware limit — GPU VRAM being exhausted or heavily contended.

**Root cause:** `ThreadPoolExecutor(max_workers=4)` in `api/server.py` allows up to 4 concurrent pipeline threads. Each thread can call Ollama simultaneously with different models. A single function run triggers **4 model swaps**:

```
deepseek-r1:8b  →  llama3.1:8b  →  qwen2.5-coder:7b  →  mistral:7b
(preprocessing      (validator)    (recommender+verify)   (reporter)
 + RAG)
```

When two threads hit different stages simultaneously, Ollama tries to load both models into VRAM → OOM error or extreme swapping slowdown.

**Fix — `pipeline/llm/ollama_client.py`:**

Added a **process-wide `threading.Semaphore`** (`_GPU_LOCK`) held for the **entire streaming duration**:

```python
_GPU_LOCK = threading.Semaphore(max_concurrent_llm_calls)

with _get_gpu_lock():          # blocks until GPU is free
    stream = ollama.chat(...)
    for chunk in stream:       # GPU in use during entire generation
        ...
    stream.close()             # GPU released here — next caller unblocks
```

This guarantees only `max_concurrent_llm_calls` models run on the GPU at any time.

**Fix — `config.yaml` (two new settings):**
```yaml
settings:
  max_concurrent_llm_calls: 1  # raise only if VRAM can hold multiple models
  model_keep_alive: 60         # seconds before Ollama unloads idle model
```

`keep_alive=60s` means:
- Retries within the same stage reuse the already-loaded model (fast)
- After 60s of inactivity the model unloads, freeing VRAM cleanly for the next model to load

**Verified:** Two threads calling `ollama_generate` concurrently — thread B waits at `"Waiting for GPU lock…"` until thread A calls `stream.close()`, then proceeds.

## Architecture reminder — model assignments

| Stage | Model | Why |
|-------|-------|-----|
| preprocessing | `deepseek-r1:8b` | Chain-of-thought for security-aware descriptions |
| rag_analyzer | `deepseek-r1:8b` | Complex CVE pattern matching — reasoning required |
| validator | `llama3.1:8b` | Fast YES/NO confirmation — no reasoning needed |
| recommender | `qwen2.5-coder:7b` | Code generation — syntax-aware |
| verifier | `qwen2.5-coder:7b` | Same model as recommender — no model swap on verify |
| reporter | `mistral:7b` | Summarisation |

**Do not swap `llama3.1:8b` into the RAG stage** — the complex multi-CVE JSON prompt causes repeated parse failures and burns all retries.
